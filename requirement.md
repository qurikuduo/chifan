# 家庭用餐协作网站 — 需求文档

## 一、项目概述

一个家庭用餐协作网站，核心流程：厨师创建菜单 → 家人选菜 → 厨师看结果做菜 → 通知开饭。

- 界面语言：**多语言支持**（中文、English、Español、العربية），自动检测浏览器语言，可手动切换并记住偏好
- 设计理念：**移动端优先**的响应式设计（手机端体验优先，同时兼容电脑浏览器）
- 支持 **PWA**（渐进式 Web 应用），手机可"添加到主屏幕"像原生 App 一样使用，支持离线缓存（API 响应的 RuntimeCaching）和推送通知
- 首页及个人资料页显示当前应用**版本号**

## 二、用户系统

### 2.1 角色与权限

| 角色 | 说明 |
|------|------|
| **管理员** | 系统初始化时自动创建一个。可以：管理所有用户、审批注册、分配家庭角色、管理系统数据（原材料、烹饪方式、标签等）。管理员本身也是家庭成员，可以选菜和创建菜单 |
| **家庭成员** | 注册后由管理员审批通过。每人设置一个家庭角色标签（预置常用标签：父亲、母亲、爷爷、奶奶、姑姑、姐姐、弟弟等，同时支持自定义输入如"小姨"、"表哥"） |

- **厨师是临时行为**：任何家庭成员都可以创建菜单（充当厨师），不需要特别授权

### 2.2 注册与登录

- **登录方式**：用户名+密码 或 邮箱+密码
- **注册流程**：用户填写用户名、邮箱、密码 → 提交注册 → 管理员审批 → 审批通过后即可登录（不需要邮箱验证）
- **管理员添加用户**：管理员可以直接创建用户账号（跳过审批）
- **忘记密码**：由管理员直接重置密码，不做邮箱重置
- **修改密码**：登录后可在个人中心自行修改密码

### 2.3 个人资料

- 用户可修改**昵称**和**家庭角色**标签
- 个人资料页展示用户基本信息与应用版本号

### 2.4 饮食偏好与过敏原

- 每位家庭成员可设置自己的**饮食备注**（自由文本，如"不吃辣"、"素食"）
- 可标记自己的**过敏食材**（从原材料库中多选，支持搜索）
- 家人偏好总览：查看所有家庭成员的饮食备注和过敏食材
- 菜单中包含家人过敏食材的菜品会显示**过敏原警告**

## 三、菜品管理（菜品库）

菜品库是所有菜品的档案库，创建菜单时从这里选菜，也可以随时新增。

### 3.1 菜品属性

| 字段 | 必填 | 说明 |
|------|------|------|
| 菜品名称 | 是 | 如"红烧牛肉"、"味增汤" |
| 菜品照片 | 否 | 可上传多张，可设置其中一张为**默认照片**；如果没有上传任何照片，使用系统通用占位图 |
| 原材料 | 否 | **多选**，从原材料库中选择（支持拼音首字母/全拼音/汉字模糊搜索） |
| 烹饪方式 | 否 | **多选**，从烹饪方式库中选择 |
| 标签(Tag) | 否 | **多选**，支持自定义新增标签。预置：中餐、西餐、西班牙菜、日本菜、汤类、炒饭、甜品等 |
| 描述/备注 | 否 | 支持 **Markdown 富文本**（粗体、斜体、标题、列表等），可在描述中嵌入图片（通过编辑器工具栏上传），前端使用 `marked` + `DOMPurify` 安全渲染 |

### 3.2 原材料管理

- 独立的原材料库，支持增删改查
- 搜索支持：拼音首字母、全拼音、汉字模糊匹配（菜品同样支持拼音搜索）
- 原材料分类：肉类、蔬菜、海鲜、调料、主食、豆制品、蛋奶等（分类可管理）

### 3.3 烹饪方式管理

- 独立的烹饪方式库，支持增删改查
- 预置：蒸、煮、煎、炸、烤、炒、拌、炖、焖、烧、卤等

### 3.4 标签管理

- 独立的标签库，支持增删改查
- 预置常用标签，支持自定义新增
- 菜品列表页可根据标签筛选

### 3.5 菜品照片规则

- 创建菜品时可上传照片
- 一个菜品可以有多张历史照片（每次创建菜单时上传的都会保留）
- 可以设置某张为**默认照片**（在菜单中展示用）
- 如果没有上传任何照片，使用系统**通用占位图**
- 照片限制：单张最大 **5MB**，格式限 **JPG/PNG/WebP**
- 上传时验证文件**魔术字节**（Magic Bytes），防止伪造文件类型
- 照片存储在 Cloudflare R2（Cloudflare 部署）或本地文件系统 `data/photos/`（Docker 自托管部署）
- 支持在菜品描述的 Markdown 中直接**嵌入图片**（通过编辑器工具栏上传）

## 四、菜单（菜肴列表）核心流程

### 4.1 创建菜单

任何家庭成员（厨师）可以创建一个菜单：

| 字段 | 必填 | 说明 |
|------|------|------|
| 菜单标题 | 是 | 如"今天的晚餐"、"周末午餐" |
| 餐次类型 | 是 | 早餐 / 午餐 / 晚餐 / 下午茶 / 宵夜 |
| 用餐时间 | 是 | 日期+时间，如"2026-03-14 18:00" |
| 菜品列表 | 是 | 从菜品库选择，也可以临时新增菜品（自动保存到菜品库） |
| 选菜截止时间 | 是 | 超过此时间无法再选菜/改选 |
| 邀请的家人 | 是 | 可选择邀请哪些家庭成员（可全选也可单独选） |

### 4.2 多人协作菜单

- 菜单创建者可以邀请其他家庭成员成为"**协作厨师**"
- 协作厨师可以往菜单里添加菜品
- 协作方式：刷新页面看到最新内容即可，无需实时 WebSocket 同步

### 4.3 自动保存

- 每添加一个菜品到菜单中，立即自动保存到服务端
- 保证网络故障时最多丢失最后一个未保存的菜品
- UI 上显示保存状态（"已保存" / "保存中..." / "保存失败，请重试"）

### 4.4 菜单状态流转

```
草稿(创建中) → 已发布(选菜中) → 选菜结束(超时或全员已选) → 烹饪中 → 已完成(开饭)
```

- **草稿**：厨师可以先建菜单但不发布，等准备好了再发布
- **已发布**：发布后被邀请的家人收到通知，可以开始选菜
- **选菜结束**：截止时间到达 或 所有被邀请的人都已选完
- **烹饪中**：厨师标记开始做菜
- **已完成**：厨师点击"饭做好了"，通知所有被邀请的家人

### 4.5 家人选菜

- 菜单发布后，被邀请的家人收到通知
- 家人打开菜单，看到所有菜品（含照片、名称、描述）
- **简单勾选**：勾选自己想吃的菜品（可多选）
- 选菜过程**公开透明**：可以看到其他家人选了什么
- 截止时间前可以修改选择
- 截止时间后，锁定选择，不可修改

### 4.6 选菜结果与打印

- 厨师可以查看选菜汇总：每个菜品被多少人选了、分别是谁
- 按受欢迎程度（选择人数）排序
- 可以打印菜单：独立的**打印预览页面**，表格形式展示菜品列表 + 选择人数 + 烹饪方式 + 原材料清单

### 4.7 完成与通知

- 厨师点击"饭做好了" → 菜单状态变为"已完成" → 通知所有被邀请的家人
- **通知方式**：站内通知（页面内通知小红点/弹窗）+ 浏览器推送通知（Push Notification）

## 五、历史与记录

- 查看历史菜单列表（按日期浏览过去的每顿饭）
- 查看某个菜品的历史被选次数 / 受欢迎程度
- **我的最爱**：根据选菜历史自动统计个人最常选择的菜品排行
- **家人最爱**：查看每位家庭成员的菜品偏好排行
- 统计功能（后续迭代加入）：最受欢迎的菜品排行榜、每个人的偏好分析

## 六、技术架构

项目支持两种部署模式：**Cloudflare 托管**和 **Docker 自托管**。

### 6.1 Cloudflare 部署架构

| 组件 | Cloudflare 服务 | 用途 |
|------|----------------|------|
| 前端 | Cloudflare Pages | 托管前端 SPA（Vue 3 + Vite） |
| 后端 API | Cloudflare Workers | 无服务器 API |
| 数据库 | Cloudflare D1 (SQLite) | 存储用户、菜品、菜单等数据 |
| 文件存储 | Cloudflare R2 | 存储菜品照片 |
| 通知 | 轮询 (Polling) | 简单可靠的通知轮询机制 |
| 认证 | Workers 内置 JWT | 用户认证与会话管理 |

### 6.2 Docker 自托管架构

| 组件 | 技术 | 用途 |
|------|------|------|
| 前端 | Vite 构建产物由后端静态托管 | SPA 资源 |
| 后端 API | Node.js 20 + Hono.js + tsx | TypeScript HTTP 服务 |
| 数据库 | better-sqlite3（D1 兼容适配器） | 本地 SQLite 文件存储 |
| 文件存储 | FileSystemStorage（R2 兼容适配器） | 本地 `data/photos/` 目录 |
| 容器化 | 多阶段 Docker 构建（node:20-alpine） | 生产部署 |
| 编排 | docker-compose.yml | 单命令启动 |

### 6.3 前端技术栈

- **Vue 3** + Vite + Pinia 状态管理 + Vue Router
- **vue-i18n** 国际化（中文/英文/西班牙文/阿拉伯文）
- 响应式移动端优先设计
- PWA 支持（VitePWA 插件、离线缓存、RuntimeCaching、添加到主屏幕）
- **Markdown 编辑器**组件（支持实时预览、图片上传）
- Markdown 渲染：`marked` + `DOMPurify`（XSS 安全）
- **pinyin-pro** 拼音搜索支持
- 构建优化：Rollup 手动分块（vendor-pinyin、vendor-markdown、vendor-vue、vendor-i18n）

### 6.4 后端技术栈

- **Hono.js** 4.6 HTTP 框架（TypeScript）
- `@hono/node-server` 用于 Docker 部署，Cloudflare Workers 入口用于云部署
- better-sqlite3 用于本地开发和 Docker 部署，Cloudflare D1 用于云部署
- JWT 认证（HS256，7 天有效期）+ 密码哈希（PBKDF2 100000 次迭代）
- 轮询方式实现通知
- **API 限流中间件**（基于内存的滑动窗口，支持多级限流预设）
- **OpenAPI 3.0 规范** + **Swagger UI**（`/api/v1/docs/`）

### 6.5 CI/CD

- **GitHub Actions**（`.github/workflows/docker.yml`）
  - 自动运行后端 + 前端单元测试 + E2E 测试
  - 测试覆盖率报告（上传 Artifact + GITHUB_STEP_SUMMARY）
  - 自动构建 Docker 镜像并推送至 **GHCR**（`ghcr.io/qurikuduo/chifan:main`）
- Cloudflare Pages 自动部署（`family-menu-jdx.pages.dev`）

### 6.6 测试

- **Vitest** 单元测试：后端 291 项 + 前端 108 项
- **Playwright** E2E 测试：30 项
- 总计 **429** 项自动化测试
