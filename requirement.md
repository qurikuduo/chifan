# 家庭用餐协作网站 — 需求文档

## 一、项目概述

一个家庭用餐协作网站，核心流程：厨师创建菜单 → 家人选菜 → 厨师看结果做菜 → 通知开饭。

- 界面语言：**纯中文**
- 设计理念：**移动端优先**的响应式设计（手机端体验优先，同时兼容电脑浏览器）
- 支持 **PWA**（渐进式 Web 应用），手机可"添加到主屏幕"像原生 App 一样使用，支持离线缓存和推送通知

---

## 二、用户系统

### 2.1 角色与权限

| 角色 | 说明 |
|------|------|
| **管理员** | 系统初始化时自动创建一个。可以：管理所有用户、审批注册、分配家庭角色、管理系统数据（原材料、烹饪方式、标签等）。管理员本身也是家庭成员，可以选菜和创建菜单 |
| **家庭成员** | 注册后由管理员审批通过。每人设置一个家庭角色标签（预置常用标签：父亲、母亲、爷爷、奶奶、姑姑、姐姐、弟弟等，同时支持自定义输入如"小姨"、"表哥"） |

- **厨师是临时行为**：任何家庭成员都可以创建菜单（充当厨师），不需要特别授权

### 2.2 注册与登录

- **登录方式**：用户名+密码 或 邮箱+密码
- **注册流程**：用户填写用户名、邮箱、密码 → 提交注册 → 管理员审批 → 审批通过后即可登录（不需要邮箱验证）
- **管理员添加用户**：管理员可以直接创建用户账号（跳过审批）
- **忘记密码**：由管理员直接重置密码，不做邮箱重置

## 三、菜品管理（菜品库）

菜品库是所有菜品的档案库，创建菜单时从这里选菜，也可以随时新增。

### 3.1 菜品属性

| 字段 | 必填 | 说明 |
|------|------|------|
| 菜品名称 | 是 | 如"红烧牛肉"、"味增汤" |
| 菜品照片 | 否 | 可上传多张，可设置其中一张为**默认照片**；如果没有上传任何照片，使用系统通用占位图 |
| 原材料 | 否 | **多选**，从原材料库中选择（支持拼音首字母/全拼音/汉字模糊搜索） |
| 烹饪方式 | 否 | **多选**，从烹饪方式库中选择 |
| 标签(Tag) | 否 | **多选**，支持自定义新增标签。预置：中餐、西餐、西班牙菜、日本菜、汤类、炒饭、甜品等 |
| 描述/备注 | 否 | 自由文本，如"微辣"、"孩子爱吃"、"需要提前腌制" |

### 3.2 原材料管理

- 独立的原材料库，支持增删改查
- 搜索支持：拼音首字母、全拼音、汉字模糊匹配
- 原材料分类：肉类、蔬菜、海鲜、调料、主食、豆制品、蛋奶等（分类可管理）

### 3.3 烹饪方式管理

- 独立的烹饪方式库，支持增删改查
- 预置：蒸、煮、煎、炸、烤、炒、拌、炖、焖、烧、卤等

### 3.4 标签管理

- 独立的标签库，支持增删改查
- 预置常用标签，支持自定义新增
- 菜品列表页可根据标签筛选

### 3.5 菜品照片规则

- 创建菜品时可上传照片
- 一个菜品可以有多张历史照片（每次创建菜单时上传的都会保留）
- 可以设置某张为**默认照片**（在菜单中展示用）
- 如果没有上传任何照片，使用系统**通用占位图**
- 照片限制：单张最大 **5MB**，格式限 **JPG/PNG/WebP**
- 照片存储在 Cloudflare R2

## 四、菜单（菜肴列表）核心流程

### 4.1 创建菜单

任何家庭成员（厨师）可以创建一个菜单：

| 字段 | 必填 | 说明 |
|------|------|------|
| 菜单标题 | 是 | 如"今天的晚餐"、"周末午餐" |
| 餐次类型 | 是 | 早餐 / 午餐 / 晚餐 / 下午茶 / 宵夜 |
| 用餐时间 | 是 | 日期+时间，如"2026-03-14 18:00" |
| 菜品列表 | 是 | 从菜品库选择，也可以临时新增菜品（自动保存到菜品库） |
| 选菜截止时间 | 是 | 超过此时间无法再选菜/改选 |
| 邀请的家人 | 是 | 可选择邀请哪些家庭成员（可全选也可单独选） |

### 4.2 多人协作菜单

- 菜单创建者可以邀请其他家庭成员成为"**协作厨师**"
- 协作厨师可以往菜单里添加菜品
- 协作方式：刷新页面看到最新内容即可，无需实时 WebSocket 同步

### 4.3 自动保存

- 每添加一个菜品到菜单中，立即自动保存到服务端
- 保证网络故障时最多丢失最后一个未保存的菜品
- UI 上显示保存状态（"已保存" / "保存中..." / "保存失败，请重试"）

### 4.4 菜单状态流转

```
草稿(创建中) → 已发布(选菜中) → 选菜结束(超时或全员已选) → 烹饪中 → 已完成(开饭)
```

- **草稿**：厨师可以先建菜单但不发布，等准备好了再发布
- **已发布**：发布后被邀请的家人收到通知，可以开始选菜
- **选菜结束**：截止时间到达 或 所有被邀请的人都已选完
- **烹饪中**：厨师标记开始做菜
- **已完成**：厨师点击"饭做好了"，通知所有被邀请的家人

### 4.5 家人选菜

- 菜单发布后，被邀请的家人收到通知
- 家人打开菜单，看到所有菜品（含照片、名称、描述）
- **简单勾选**：勾选自己想吃的菜品（可多选）
- 选菜过程**公开透明**：可以看到其他家人选了什么
- 截止时间前可以修改选择
- 截止时间后，锁定选择，不可修改

### 4.6 选菜结果与打印

- 厨师可以查看选菜汇总：每个菜品被多少人选了、分别是谁
- 按受欢迎程度（选择人数）排序
- 可以打印菜单：菜品列表 + 选择人数 + 原材料清单

### 4.7 完成与通知

- 厨师点击"饭做好了" → 菜单状态变为"已完成" → 通知所有被邀请的家人
- **通知方式**：站内通知（页面内通知小红点/弹窗）+ 浏览器推送通知（Push Notification）

## 五、历史与记录

- 查看历史菜单列表（按日期浏览过去的每顿饭）
- 查看某个菜品的历史被选次数 / 受欢迎程度

## 六、技术架构（基于 Cloudflare）

| 组件 | Cloudflare 服务 | 用途 |
|------|----------------|------|
| 前端 | Cloudflare Pages | 托管前端 SPA（Vue 3 + Vite） |
| 后端 API | Cloudflare Workers | 无服务器 API |
| 数据库 | Cloudflare D1 (SQLite) | 存储用户、菜品、菜单等数据 |
| 文件存储 | Cloudflare R2 | 存储菜品照片 |
| 通知 | 轮询 (Polling) | 简单可靠的通知轮询机制 |
| 认证 | Workers 内置 JWT | 用户认证与会话管理 |

### 前端技术栈

- **Vue 3** + Vite
- 响应式移动端优先设计
- PWA 支持（Service Worker、离线缓存、推送通知、添加到主屏幕）

### 后端技术栈

- Cloudflare Workers（TypeScript）
- Cloudflare D1 作为关系型数据库
- Cloudflare R2 存储图片
- JWT 做认证
- 轮询方式实现通知
